Shopify軽量化

Shopify軽量化のベストプラクティス

Shopifyの表示速度は、画像・JavaScript・計測方法の3点を先に整えると改善しやすいです。

Shopify公式は、まずJavaScriptを減らし、HTMLとCSSを基本に組むことを強く勧めています。

画像は「必要な大きさで出す」「画面外は遅延読み込みする」だけでも、体感速度に効きやすいです。

改善は感覚で決めず、Core Web Vitalsという実ユーザー指標で継続監視するのが安全です。

1. 最初に見るべきは「速いかどうか」ではなく、何を測るか

Googleの web.dev では、Web Vitals はサイト体験を共通ルールで測る指標群だと整理されています。特に Core Web Vitals は、読み込み、操作反応、画面のズレを確認するための重要指標です。

目安としては、LCP は 2.5 秒以内、INP は 200 ミリ秒以下、CLS は 0.1 以下が「良い」とされています。LCP は「メインの見出しや大きな画像が見えるまでの速さ」、INP は「触ったときの反応速度」、CLS は「読み込み中にレイアウトがずれる量」です。

要するに、Shopify軽量化は「なんとなく速く見える」ではなく、まず測る物差しを決めるところから始めるのが実務的です。

2. Shopify公式が最優先にしているのは、JavaScriptを増やしすぎないこと

Shopify公式ドキュメントは、テーマの基本機能は HTML と CSS を中心に作り、JavaScript は補助的に使うべきだと明記しています。商品を探す、購入するといった基本動作に JavaScript が必須な設計は避けるべき、という考え方です。

同じページでは、圧縮後の JavaScript バンドルは 16KB 以下を目安にするよう案内されています。これは「読み込むスクリプトの塊を小さくする」という意味で、特にトップページや商品ページで効きやすいです。

また、使わない部品や重い機能は最初から全部読むのではなく、必要になったタイミングで読み込む形に寄せると、初回表示が軽くなります。

3. 画像は「きれいに見せる」より「適切な大きさで出す」が先

Shopify公式は、レスポンシブ画像、つまり端末サイズに応じて適切な画像サイズを返す仕組みを推奨しています。大きすぎる画像を小さなスマホにそのまま送ると、無駄な通信が増えて表示が遅くなります。

加えて、最初の画面に見えていない画像は `loading='lazy'` で遅延読み込みにするよう案内されています。これは「すぐ見えない画像は後回しにする」設定です。ただし、最初に見える大事な画像まで遅延読み込みすると逆効果なので、ファーストビューの画像には使わないのが基本です。

さらに Shopify の CDN は画像や静的ファイルの配信を自動最適化してくれますが、その恩恵を受けるには、外部にばらまかず、できるだけ Shopify 側の配信基盤に載せるのが有利です。

4. Liquid と CSS も積み重なると遅くなる

Shopify公式は、Liquid の複雑な処理をループのたびに繰り返すと描画時間が伸びると説明しています。Liquid は Shopify テーマで使うテンプレート言語で、商品一覧や価格表示の出し分けを書く部分です。

例えば、同じ並び替えや計算を商品ごとに何度も回すのではなく、先にまとめて処理してから表示する方が軽くなります。あわせて、CSS は必要な画面だけに読み込まれる構成に寄せると、不要な読み込みを減らせます。

つまり、アプリや画像だけでなく、テーマの中の書き方そのものも速度に効く、という理解が必要です。

5. 改善は Lighthouse だけで終わらせず、実ユーザー指標で追い続ける

Shopifyテーマの公式ガイドでは、ホーム、商品、コレクションの主要ページを基準に Lighthouse で確認する流れが紹介されています。一方で web.dev は、Core Web Vitals は本来「実際の利用者の体験」で見るべきで、Lighthouse のような実験室データだけでは不十分だと説明しています。

そのため、改善の実務は「Lighthouse で原因を探す」「実ユーザーデータで成果を確認する」の二段構えが安全です。特にECでは、見た目の派手さより、商品画像が早く出ること、カート操作が引っかからないこと、読み込み中にボタン位置がずれないことが売上に直結しやすいです。

最短で効果を出すなら、1. 主要ページのLCP悪化要因を特定する、2. 不要なJavaScriptを減らす、3. 画像サイズと遅延読み込みを整理する、の順が再現性の高い進め方です。
